# PR Review Cycle — app_badge_control_flutter 리뷰 자동화 오케스트레이터

이 스킬은 Flutter **플러그인** 저장소(app_badge_control_flutter)에 맞춰 구성되어 있다.
버전 브랜치·rebase·changelog·fastlane 같은 앱 배포 절차는 **다루지 않는다**. 리뷰 코멘트 제출 방식과
블라인드 서브에이전트 판단 사이클에만 집중한다.

## 입력 형식

```
/pr_review [<pr-number>]
```

- `<pr-number>` 생략 시: 현재 브랜치에 연결된 열린 PR을 자동으로 찾는다
  (`gh pr view --json number -q .number`).
- 명시 시: 해당 PR을 대상으로 한다.

---

## Phase 0: 저장소 · PR 확인

```bash
# 레포 정보
gh repo view --json owner,name -q '"\(.owner.login)/\(.name)"'   # 예: toyaji/app_badge_control_flutter

# 대상 PR 번호 (인자 없으면 현재 브랜치 PR)
gh pr view <pr-number|생략> --json number,headRefName,baseRefName,headRefOid
```

- `headRefOid`를 리뷰 제출용 `commit_id`로 저장한다.
- 이 저장소는 iOS/Android/macOS/Windows/Web 5개 플랫폼을 지원하는 플러그인이다. 리뷰 시
  **플랫폼 간 동작 일관성**을 최우선 관점으로 둔다.

## Phase 1: 코드 리뷰 코멘트 제출 (Auditor 역할 — 오케스트레이터가 직접 수행)

> ⚠️ 이 단계는 리뷰를 작성하는 주체(오케스트레이터)가 수행한다. 이후 Phase 3의 판단은
> 이 사고 과정에 **블라인드인 서브에이전트**가 맡는다. 그래야 자기 리뷰에 대한
> 확증 편향 없이 수용/반박이 판정된다.

### 1-1. 시작 전 준비 — 가이드라인 숙지

diff를 보기 전에 아래를 먼저 읽어 리뷰 기준을 세운다:

```
Read: .agents/AGENTS.md      # 커밋/테스트/lint 규칙
Read: README.md              # 플랫폼별 동작·제약 명세 (문서 정확성 리뷰 기준)
```

### 1-2. Diff 정독

```bash
git diff origin/<base-branch>...origin/<head-branch>
```

- diff에 등장하는 **모든 파일을 처음부터 끝까지** 읽는다.
- 변경 함수의 호출부·정의부·연관 테스트를 추적한다.
- 코멘트 하나를 쓰기 전에 관련 코드를 충분히 읽어 이슈를 **100% 확정**한다.
- **코멘트 수보다 정확성이 중요하다. 확신 없는 코멘트는 달지 않는다.**

### 1-3. 이 플러그인의 중점 리뷰 관점

| 관점 | 확인 사항 |
|------|-----------|
| **플랫폼 동작 일관성** | 같은 API 호출(`updateBadgeCount(0)`, `removeBadge()` 등)이 iOS/Android/macOS/Windows/Web에서 동일하게 동작하는가 |
| **메서드 채널 계약** | 네이티브 채널 이름·메서드명·인자 타입이 Dart(`method_channel`)와 정확히 일치하는가 |
| **네이티브 리소스 관리** | (Windows/GDI) HICON·HBITMAP·HDC·브러시/펜/폰트 누수, COM 초기화 스레드; (Swift) 메인 스레드 요구 API |
| **에러 처리** | 예외를 조용히 삼켜 호출자가 실패를 알 수 없게 되는가 (의도된 무음 처리라면 근거 확인) |
| **문서 정확성** | README/pubspec 설명이 실제 지원 플랫폼·제약(HTTPS, 브라우저 지원 범위 등)과 일치하는가 |

### 1-4. 단일 리뷰 블록으로 전체 코멘트 제출 (필수 준수)

**⚠️ 핵심: 모든 인라인 코멘트는 반드시 GitHub Reviews API를 통해 한 번의 호출로 제출한다.**
개별 `pulls/comments` 반복 호출은 UI에서 코멘트마다 별도 리뷰 블록이 생겨 보기 불편하므로 **절대 사용하지 않는다.**

이스케이프 문제를 피하기 위해 각 코멘트 본문을 파일로 작성한 뒤 `jq`로 payload를 조립하고 `--input`으로 제출한다:

```bash
COMMIT_ID=$(gh pr view <pr-number> --json headRefOid -q .headRefOid)

jq -n --arg commit_id "$COMMIT_ID" \
  --rawfile summary body_summary.md \
  --rawfile c1 body_1.md \
  --rawfile c2 body_2.md \
  '{
    commit_id: $commit_id,
    event: "COMMENT",
    body: $summary,
    comments: [
      {path: "<파일경로>", line: <라인>, side: "RIGHT", body: $c1},
      {path: "<파일경로>", line: <라인>, side: "RIGHT", body: $c2}
    ]
  }' > payload.json

gh api repos/<repo>/pulls/<pr-number>/reviews --method POST --input payload.json
```

- **인라인 코멘트는 diff에 포함된 라인(추가/변경된 라인)에만 달 수 있다.** 신규 파일은 모든 라인이
  대상이 된다. 미변경 라인에 대한 지적은 인라인이 아니라 **리뷰 요약(body)**에 서술한다.
- `side: "RIGHT"` = 변경 후 파일 기준.

### 1-5. 코멘트 작성 가이드

#### 우선순위 체계

| 등급 | 아이콘 | 기준 | 대응 |
|------|--------|------|------|
| P0 | 🔴 | 크래시·데이터 유실·빌드 실패 등 즉시 장애 | 머지 전 필수 수정 |
| P1 | 🟡 | 동작 회귀·플랫폼 간 불일치·에러 삼키기 등 명확한 버그 | 머지 전 필수 수정 |
| P2 | ⚪️ | 문서 부정확·가독성·메타데이터·테스트 누락 등 잠재 위험/구조 문제 | 권고 |

#### 코멘트 형식 — 이슈 복잡도에 맞게 선택

모든 코멘트가 같은 형식일 필요는 없다. **이슈가 단순하면 짧게 쓴다.** 억지로 항목을 채우지 않는다.

**단순한 이슈** — 간결하게:
```markdown
### ⚪️ P2 [문서] — 지원 브라우저 범위 명시 필요

Web Badging API는 Chrome/Edge만 지원. README 표에 지원 브라우저를 명시.
```

**복잡한 이슈** (재현 흐름이 필요한 경우) — 구조화해서:
```markdown
### 🟡 P1 [상태 불일치] — `updateBadgeCount(0)`이 배지를 지우지 않고 "0"을 표시

**위치:** `파일` · `함수()`

문제 설명 (핵심 원인 **굵게**, `변수/클래스`는 인라인 코드).

**재현 흐름:** A → B → C
**수정 방향:** ...
```

**작성 원칙:**
- 제목은 `### 아이콘 Px [카테고리] — 제목` 형식으로 고정.
- 카테고리 예: `버그`, `상태 불일치`, `에러 삼키기`, `리소스 누수`, `문서`, `가독성`, `메타데이터`, `테스트 누락`.
- `재현 흐름`·`수정 방향`은 인과가 복잡할 때만 포함. 단순 이슈에 억지로 채우지 않는다.
- 코드 스니펫은 문제를 명확히 드러낼 때만, 길면 핵심 줄만 인용.

### 1-6. ⚠️ 사실 기반 리뷰 원칙 (필수 준수)

**코멘트는 "이 코드 라인이 이 이유로 잘못됐다"는 사실만 담는다. 가능성·의혹·추측은 코멘트가 아니다.**

**절대 금지 — 아래 표현이 하나라도 포함되면 그 코멘트는 제출하지 않는다:**
- 가능성: "~일 수 있습니다", "~가능성이 있습니다"
- 위임: "확인이 필요합니다", "확인해보세요", "검토 필요"
- 불확실: "~인지 모르겠습니다", "명확하지 않습니다"
- 추측: "~인 것 같습니다", "~처럼 보입니다"

**올바른 코멘트 구조:** ①무엇이 잘못됐는지 단언 → ②어느 파일 몇 번째 줄이 근거인지 명시 → ③실행 시 어떤 오동작이 발생하는지 구체적으로.

### 1-7. ⛔ 이슈 없으면 없다고 하라 — 가장 중요한 원칙

- 코드가 올바르면 코멘트를 달지 않는다. 이슈 수를 채우려 억지 P2를 만들지 않는다.
- 리뷰 요약에 "전반적으로 깔끔합니다" 같은 칭찬 padding을 넣지 않는다. 발견한 이슈만 서술.
- 이슈가 하나도 없으면 인라인 없이 요약에 "검토 결과 지적 사항 없음" 한 줄로 끝낸다.
- **코멘트 수는 품질 지표가 아니다. 0개가 정답일 때도 있다.**

제출 후 `review_id`와 코멘트 수를 기록한다.

## Phase 2: 코멘트 수집

블라인드 판단을 위해 서브에이전트에 넘길 코멘트를 수집한다:

```bash
# 인라인 코멘트
gh api repos/<repo>/pulls/<pr-number>/comments \
  --jq '[.[] | {id: .id, path: .path, line: .line, body: .body}]'

# 리뷰 요약(레벨) 코멘트
gh api repos/<repo>/pulls/<pr-number>/reviews \
  --jq '[.[] | {id: .id, body: .body, state: .state}]'
```

## Phase 3: 판단·수정·검증·답변·Resolve (pr-review-resolver 서브에이전트)

**리뷰어(오케스트레이터)의 사고에 블라인드인 `pr-review-resolver` 서브에이전트를 호출한다.**
서브에이전트는 새 컨텍스트에서 시작하므로 오케스트레이터의 리뷰 근거·의도를 알지 못한다.
코멘트와 실제 코드만 보고 **독립적으로** 수용/반박을 판정하게 하는 것이 핵심이다.

> ⛔ **오케스트레이터 필독 — 절대 금지**: 서브에이전트에 전달하는 프롬프트에서
> **"커밋 & 푸시 — 사용자 승인 후 진행" 단계를 수정하거나 자동화하지 않는다.**
> "오케스트레이터가 대신 승인", "바로 커밋", "승인 없이 진행" 같은 대체는 엄격히 금지한다.
> 이 단계는 반드시 사용자가 직접 승인해야 한다. (`.agents/AGENTS.md` 커밋 규칙과 일치)

**서브에이전트에 전달하는 프롬프트 템플릿:**

```
PR #<pr-number> (<repo>, 브랜치: <head-branch>)의 리뷰 코멘트를 처리해주세요.

작업 디렉토리: <cwd>
현재 브랜치: <head-branch>

당신은 이 리뷰를 작성하지 않았습니다. 리뷰어의 판단을 그대로 믿지 말고,
아래 코멘트와 실제 코드를 직접 읽어 수용/반박을 스스로 결정하세요.

## 리뷰 코멘트 목록 (JSON)
<Phase 2에서 수집한 코멘트 JSON 붙여넣기>

작업 지침은 당신의 에이전트 정의(pr-review-resolver)를 따르세요.
완료 후 코멘트 ID별 판단 결과·수정 파일·검증 결과를 보고하세요.
```

서브에이전트(`pr-review-resolver`)가 수행하는 일:
1. `.agents/AGENTS.md` 숙지
2. 각 코멘트에 대해 코드를 직접 읽고 **수용 / 부분 수용 / 반박** 독립 판정
3. 수용분 코드 수정 (최소 변경 원칙)
4. 검증: `flutter analyze` (경고 0), `flutter test` (관련 테스트)
5. **커밋 & 푸시는 사용자 승인 후에만**
6. 각 코멘트에 답변 → 스레드 resolve

## 에러 처리

| 상황 | 처리 |
|------|------|
| 인라인 코멘트 라인이 diff 밖 (422) | 해당 지적을 리뷰 요약(body)으로 옮겨 제출 |
| PR 없음 (현재 브랜치) | 사용자에게 PR 번호를 요청하거나 PR 생성 여부 확인 |
| 서브에이전트 실패 | 1회 재시도, 재실패 시 해당 코멘트를 "미처리"로 보고 후 계속 |
| GraphQL resolve 실패 | 답변만 달고 resolve 실패 목록을 사용자에게 보고 |
